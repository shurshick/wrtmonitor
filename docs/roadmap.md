# Roadmap

Этот документ фиксирует будущие этапы развития `wrtmonitor`. Порядок отражает желаемую последовательность: сначала стабилизировать базовый контур, затем расширять управление роутером и только после этого добавлять более сложные сценарии вроде локального поиска.

## Этап 1. Стабилизация тестовой версии

Цель: довести текущий контур до состояния, в котором его можно уверенно тестировать на реальном сервере и OpenWrt-роутере.

Задачи:

- проверить установку сервера через TrueNAS Custom App;
- проверить установку сервера через обычный Docker Compose;
- проверить первичную настройку `/setup`;
- проверить вход Android-приложения по логину и паролю администратора;
- проверить установку OpenWrt-агента и регистрацию роутера на сервере;
- привести README, инструкции TrueNAS, Android и OpenWrt к одному актуальному сценарию;
- собрать понятный тестовый релиз с YAML, Android APK и архивом агента.

Критерий готовности:

- сервер, Android и OpenWrt-агент проходят полный ручной тест на реальном окружении.

## Этап 2. Android-приложение как рабочий клиент

Цель: заменить статические экраны реальными данными с сервера.

Задачи:

- загрузка списка роутеров с сервера;
- карточка роутера со статусом, моделью, прошивкой и временем последней связи;
- экран телеметрии;
- обработка ошибок сети и авторизации;
- обновление access token или понятный повторный вход;
- переключение языка интерфейса русский/английский;
- сохранение адреса сервера и сессии в безопасном хранилище Android;
- базовая Material 3-полировка интерфейса.

Критерий готовности:

- в приложении виден реальный подключенный роутер и его актуальное состояние.

## Этап 3. Расширенная телеметрия OpenWrt

Цель: сделать агент полезным не только как heartbeat, а как источник состояния роутера.

Задачи:

- сбор uptime, load average, памяти и диска;
- сбор состояния WAN/LAN;
- сбор IP-адресов и статуса интерфейсов;
- сбор Wi-Fi статуса через `ubus`/`wifi`;
- список подключённых клиентов;
- качество сигнала Wi-Fi клиентов, где доступно;
- версия OpenWrt, модель, hostname;
- аккуратная деградация, если команда недоступна на конкретной прошивке.

Критерий готовности:

- сервер получает структурированную телеметрию, которую можно показать в Android.

## Этап 4. Управление роутером из приложения

Цель: добавить реальные безопасные действия управления через сервер.

Задачи:

- перезагрузка роутера;
- включение и выключение Wi-Fi;
- изменение SSID;
- изменение Wi-Fi пароля;
- просмотр и управление гостевой сетью;
- просмотр сетевых интерфейсов;
- очередь команд с понятными статусами: создана, отправлена, выполнена, ошибка;
- показ результата команды в Android;
- аудит всех действий администратора.

Критерий готовности:

- основные действия выполняются через Android и подтверждаются результатом от агента.

## Этап 5. Пользователи, роли и безопасность

Цель: подготовить проект к более безопасному использованию не только в домашней лаборатории.

Задачи:

- роли пользователей: владелец, администратор, просмотр;
- смена пароля администратора;
- отключение пользователя;
- управление device token;
- отзыв токена роутера;
- журнал аудита в интерфейсе;
- ограничение опасных команд;
- защита от повторного provisioning без разрешения владельца;
- документация по HTTPS, NPM, секретам и резервному копированию.

Критерий готовности:

- владелец может контролировать доступ пользователей и подключённых роутеров.

## Этап 6. Установка и обновление

Цель: сделать установку и обновление предсказуемыми.

Задачи:

- стабильные Docker tags;
- release notes на русском;
- отдельные YAML для TrueNAS;
- инструкция обновления без потери данных PostgreSQL;
- проверка совместимости БД при обновлении;
- резервное копирование и восстановление;
- понятный rollback на предыдущий релиз;
- подготовка Android APK в релизах.

Критерий готовности:

- пользователь может обновить тестовую установку по инструкции без ручной правки внутренних файлов.

## Этап 7. Поддержка дополнительных платформ агентов

Цель: подготовить архитектуру агентов к разным типам роутеров и Linux-окружений.

Задачи:

- выделить общую модель возможностей агента;
- добавить capability report: какие команды поддерживает конкретный агент;
- рассмотреть отдельный Keenetic/Entware-агент;
- не смешивать OpenWrt UCI/ubus-логику с другими платформами;
- показывать в Android только доступные действия для конкретного роутера.

Критерий готовности:

- сервер и Android понимают, что разные агенты могут поддерживать разный набор функций.

## Этап 8. Локальный поиск и подключение OpenWrt-роутера из Android

Статус: будущий функционал, последний этап после стабилизации базового управления.

Цель: упростить первичное подключение роутера. Пользователь открывает Android-приложение, нажимает кнопку поиска, приложение находит OpenWrt-роутер в локальной сети и предлагает подключить его к уже работающему серверу `wrtmonitor`.

Исходные условия:

- сервер развёрнут и работает;
- Android-приложение установлено и подключено к серверу;
- OpenWrt-агент установлен на роутер, но может быть ещё не привязан к серверу.

Ожидаемый сценарий:

1. Пользователь нажимает в Android кнопку поиска.
2. Приложение сканирует локальную сеть телефона.
3. Найденные роутеры показываются списком.
4. Пользователь выбирает роутер и подтверждает подключение.
5. Приложение авторизуется на сервере логином и паролем владельца.
6. Сервер выдаёт device token через `/api/v1/devices/provision`.
7. Android передаёт роутеру адрес сервера и device token через локальный setup endpoint агента.
8. Агент сохраняет настройки, перезапускается и начинает работать через сервер.

Необходимые изменения:

- добавить в OpenWrt-агент локальный setup/discovery endpoint;
- добавить в Android экран поиска, список найденных устройств и flow подключения;
- добавить на сервер аудит события подключения роутера через мобильное приложение;
- продумать защиту локальной настройки: одноразовый PIN, временное окно настройки или явное включение setup-режима на роутере.

Оценка сложности:

- MVP: 3-5 рабочих дней;
- надёжная версия с нормальной безопасностью и UX: 1-2 недели.

Риски и ограничения:

- Android ограничивает сетевое сканирование и доступ к Wi-Fi-информации;
- телефон должен быть подключён к той же локальной сети, где находится роутер;
- подсеть может отличаться от `/24`, её нужно определять автоматически;
- локальный setup endpoint нельзя оставлять открытым без защиты.
