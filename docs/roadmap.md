# Roadmap

Этот документ фиксирует этапы развития `wrtmonitor`. Порядок важен: сначала стабилизируем сервер, агент, телеметрию и Android-клиент, затем расширяем управление роутером. Локальный поиск роутера из Android оставлен на последний этап.

## Уже сделано в v0.1.0-test.11

- Сервер запускается в Docker и через TrueNAS Custom App.
- PostgreSQL используется как основная база данных.
- Схема базы ведётся через Alembic migrations.
- Первичная настройка сервера создаёт администратора через `/setup`.
- Android-приложение запрашивает адрес сервера и входит по логину/паролю администратора.
- Android показывает список устройств и экран устройства.
- OpenWrt agent устанавливается через release archive.
- Агент регистрирует роутер на сервере через логин/пароль администратора.
- Агент сохраняет `server_url`, `device_id` и `device_token` в UCI.
- Агент отправляет telemetry через серверный API.
- Телеметрия собирается через `ubus` и деградирует без падения, если часть данных недоступна.
- Сервер отдаёт API последней телеметрии для Android и Web UI.
- Сервер хранит последние 100 telemetry snapshots на устройство.
- Сервер не стартует с дефолтными JWT/DB-секретами.
- Web UI `/devices` требует авторизованную сессию.
- CI проверяет backend E2E, agent smoke tests, синтаксис shell-скриптов, shellcheck, Docker build и Android build.
- В релизе публикуются TrueNAS YAML, архив OpenWrt agent, Android debug APK и `SHA256SUMS.txt`.

## Этап 1. Закрепление тестового контура

Цель: сделать текущую тестовую установку предсказуемой для повторных проверок на реальном сервере и OpenWrt.

Задачи:

- проверить установку сервера через TrueNAS Custom App на чистой системе;
- проверить обновление сервера без удаления PostgreSQL volume;
- проверить установку OpenWrt agent на реальном роутере;
- проверить обновление OpenWrt agent поверх существующей UCI-конфигурации;
- добавить больше диагностических сообщений в агент;
- улучшить страницу Web UI `/devices`, чтобы проще видеть последнюю телеметрию;
- описать rollback на предыдущий тестовый образ.

Критерий готовности: сервер, Android и OpenWrt agent проходят повторяемый ручной тест установки, обновления и передачи телеметрии.

## Этап 2. Команды управления роутером

Цель: добавить первые реальные безопасные действия управления через сервер и Android.

Задачи:

- оформить API создания команды для устройства;
- показать очередь и статус команд в Android;
- добавить результат выполнения команды;
- реализовать reboot как первую тестовую команду;
- добавить enable/disable Wi-Fi;
- добавить allowlist команд на стороне сервера и агента;
- добавить аудит действий администратора.

Критерий готовности: команда создаётся из Android, агент забирает её, выполняет и возвращает результат.

## Этап 3. Расширенная телеметрия

Цель: сделать экран устройства полезным не только как heartbeat, а как обзор состояния роутера.

Задачи:

- добавить память и свободное место;
- добавить WAN/LAN status;
- добавить IP-адреса интерфейсов;
- добавить список Wi-Fi радиомодулей;
- добавить список подключённых клиентов;
- добавить качество сигнала клиентов, где это доступно;
- сохранить аккуратную деградацию, если команда недоступна в конкретной сборке OpenWrt.

Критерий готовности: Android показывает структурированную телеметрию роутера без ручного чтения логов.

## Этап 4. Настройки Wi-Fi и сети

Цель: приблизить Android-приложение к мобильному приложению для управления домашним роутером.

Задачи:

- просмотр текущего SSID;
- изменение SSID;
- изменение Wi-Fi пароля;
- включение и выключение гостевой сети;
- просмотр сетевых интерфейсов;
- базовая настройка DHCP/адресации после отдельного анализа безопасности;
- подтверждение опасных действий перед применением.

Критерий готовности: базовые настройки Wi-Fi выполняются через Android и подтверждаются состоянием агента.

## Этап 5. Пользователи, токены и безопасность

Цель: подготовить проект к более безопасному использованию за пределами домашней лаборатории.

Задачи:

- смена пароля администратора;
- управление пользователями;
- роли: владелец, администратор, просмотр;
- отзыв device token;
- повторное provisioning только с подтверждением владельца;
- журнал аудита в интерфейсе;
- документация по HTTPS, Nginx Proxy Manager, backup и восстановлению.

Критерий готовности: владелец может контролировать доступ пользователей и подключённых роутеров.

## Этап 6. Android release build

Цель: сделать Android-сборку удобной для тестеров.

Задачи:

- подготовить signed APK;
- добавить версионирование приложения;
- добавить понятный экран обновления/проверки версии;
- улучшить хранение сессии и повторный вход;
- довести переключение языка русский/английский во всех экранах.

Критерий готовности: тестер устанавливает APK без сборки проекта вручную и понимает, какая версия стоит.

## Этап 7. Дополнительные платформы агентов

Цель: подготовить архитектуру к агентам не только для OpenWrt.

Задачи:

- выделить общую модель возможностей агента;
- добавить capability report;
- рассмотреть отдельный Keenetic/Entware agent;
- не смешивать OpenWrt UCI/ubus-логику с другими платформами;
- показывать в Android только действия, доступные конкретному агенту.

Критерий готовности: сервер и Android понимают, что разные агенты поддерживают разный набор функций.

## Этап 8. Локальный поиск роутера из Android

Статус: будущий функционал, последний этап после стабилизации базового управления.

Цель: упростить первичное подключение OpenWrt-роутера к уже работающему серверу.

Ожидаемый сценарий:

1. Пользователь открывает Android-приложение и нажимает кнопку поиска.
2. Приложение сканирует локальную сеть телефона.
3. Найденные OpenWrt-роутеры показываются списком.
4. Пользователь выбирает роутер и подтверждает подключение.
5. Приложение авторизуется на сервере логином и паролем владельца.
6. Сервер выдаёт device token через `/api/v1/devices/provision`.
7. Android передаёт роутеру адрес сервера и device token через локальный setup endpoint агента.
8. Агент сохраняет настройки, перезапускается и начинает работать через сервер.

Что понадобится:

- локальный setup/discovery endpoint в OpenWrt agent;
- экран поиска и подключения в Android;
- защита локального setup-режима: одноразовый PIN, временное окно настройки или явное включение setup-режима на роутере;
- аудит подключения на сервере.

Оценка сложности:

- MVP: 3-5 рабочих дней;
- аккуратная версия с нормальной безопасностью и UX: 1-2 недели.
